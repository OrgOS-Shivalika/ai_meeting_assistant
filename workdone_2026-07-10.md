## 1. Frontend build blocker — package-lock out of sync

### Symptom

Railway build failed at the frontend stage:

```
npm error `npm ci` can only install packages when your package.json and
package-lock.json or npm-shrinkwrap.json are in sync.
Missing: @emnapi/core@1.11.2 from lock file
Invalid: lock file's @emnapi/wasi-threads@1.2.1 does not satisfy @emnapi/wasi-threads@1.2.2
```

Build died at `[frontend 4/6] RUN npm ci`.

### Root cause

`npm ci` (used by Railway) refuses to install when the lock file
doesn't match the manifest — it's a safety property, not a bug. Someone
(or a transitive dependency's peer resolution) modified `package.json`
without updating `package-lock.json`. The `@emnapi/*` chain (a
Node-native WASM helper pulled in by `@rolldown/plugin-babel` or a
build-time binding) shipped a new patch and the lock update wasn't
committed.

### Fix (owed to the user — locally executed)

```bash
cd meeting_ai_frontend
npm install     # re-resolves and writes an updated package-lock.json
```

Then commit + push the updated lockfile:

```bash
git add meeting_ai_frontend/package-lock.json
git commit -m "chore: sync package-lock after transitive dep updates"
git push
```

Railway rebuild passes. Nothing in the Python codebase needed to
change — this was purely a frontend-toolchain hygiene issue.

### Files affected

- `meeting_ai_frontend/package-lock.json` (must be regenerated locally)

## 2. Agents v2 — architecture plan

### Context

The current agent system (`app/services/agents/…`) is a "Phase 2 hybrid":
`TranscriptAnalyzer` does ~90 % of the real AI work in one big call,
and the "agents" that follow are plugins that either duplicate what
master already produced or add narrow specialty outputs. The user
called this out — "isn't TranscriptAnalyzer doing the real work?" — and
we agreed the current abstraction is aspirational scaffolding, not a
running agent system.

The plan today: introduce a per-team first-class agent architecture
that replaces the master-analyzer-plus-plugins model with **one
addressable agent per team**, each owning its own prompt, skills, and
configuration.

### Decisions taken (recorded in the plan)

1. **Same repo as backend** — `app/agents_v2/` alongside existing code.
2. **DB row overrides win over folder manifest** — code = defaults, DB
   = per-org overrides. Admin can turn knobs without a redeploy.
3. **HR team is the pilot** — one folder, one team, one row. Prove the
   plumbing before expanding.

### Deliverable

A comprehensive plan doc lives at [AGENTS_V2_PLAN.md](AGENTS_V2_PLAN.md).
19 sections, covering:

- Directory layout for `app/agents_v2/`
- `agents_v2` table schema
- Agent contract (`manifest.py`, `config.py`, `agent.py`, `execution.py`,
  `prompts/master.md`)
- Shared library (`shared/skills`, `shared/tools`, `shared/schemas`)
- `KnowledgeContext` design (short-term facts + long-term summaries +
  open tasks in scope)
- Orchestrator behavior (route → resolve → knowledge → invoke)
- Boot lifecycle (discovery + DB seeding)
- Meeting lifecycle (end-to-end trace)
- DB override mechanics
- HR pilot setup
- Frontend impact (none day-1)
- 4-increment rollout plan
- Observability, testing, migration path

The plan file is a self-contained reference — anyone starting fresh

### 3.4 The `KnowledgeContext` schema

**File:** `app/agents_v2/shared/schemas/knowledge_context.py`

A plain dataclass — no Pydantic — that packages three memory tiers for
the agent:

- `prior_facts: list[str]` — top 10 distilled facts from
  `org_memory_facts`
- `recent_summaries: list[LongTermMeetingSummary]` — last 5 meetings in
  scope (title + when + summary)
- `open_tasks: list[OpenTask]` — up to 20 unfinished tasks in scope
- `relevant_chunks: list[str]` — reserved for a future vector-search
  pass over `meeting_chunks`

Includes a `.render_block(max_chars=3500)` method that produces a
markdown block ready to inject into the master prompt.
`is_empty()` short-circuits rendering to `""` on a fresh scope so the
prompt template can `if block: ...` without extra branches.

### 3.5 The registry

**File:** `app/agents_v2/registry.py`

At app startup, `bootstrap()`:

1. Uses `pkgutil.iter_modules` to walk `app/agents_v2/` subdirectories.
2. Skips `shared/`.
3. For each remaining folder, imports `<folder>.agent` and reads its
   `MANIFEST`.
4. Caches the module in `_AGENT_MODULES` (keyed by slug) so the
   orchestrator can invoke it later without re-importing.
5. Reads each manifest's `seed_scopes` and, for any (org, cat, team)
   tuple where no active `agents_v2` row exists, INSERTs a default row.
   Existing rows are **never overwritten** — the DB is the
   authoritative source of overrides.
6. Wrapped non-fatal — a broken agent folder logs an error but doesn't
   kill startup.

### 3.6 The orchestrator

**File:** `app/agents_v2/orchestrator.py`

Four functions, each with one responsibility:

- `has_agent_for_scope(db, meeting) -> bool` — the feature flag. Called
  by the meeting pipeline before it decides which analyzer to invoke.
- `_route(db, org, cat, team)` — SELECT the most-specific active agent
  row. Precedence: `(org, cat, team) > (org, cat) > (org)`.
- `_merge_manifest_with_row(manifest, row)` — layer DB overrides over
  manifest defaults. Empty override = use default.
- `_build_knowledge(db, meeting)` — wraps `MemoryAccess` +
  `LongTermMemory` to produce a `KnowledgeContext`. All lookups are
  try/except-wrapped so a memory-layer failure doesn't kill the run.
- `run_meeting_analysis(db, transcript, meeting)` — the public entry
  point. Routes → resolves → builds knowledge → invokes the agent
  module's `run(...)`.

The orchestrator knows almost nothing about what agents DO. Each
agent's folder owns its own execution details.

## 4. Agent scope correction — category → team

### The mistake I made

The first iteration seeded the HR agent as **category-scoped**:

```python
seed_scopes: [
  {organization_id: ..., category_id: 4554, team_id: None},
]
```

That's a category agent, not a team agent. It would match every team
inside HR. Per the agreed architecture ("team agents are the sole
executors, category and org are control planes"), this was
inconsistent with the plan.

### How it got caught

The user reviewed and asked directly: **"isn't this the category
agent?"** — and they were right. I built exactly what they typed but
didn't cross-check against the architecture we'd agreed on.

### The correction

The user provided `team_id = 3864`. Verified against the DB:

- Team `3864` = **Learning & Development**
- Sits inside category `4554` = **HR**
- Belongs to the correct org (`0dd7e275…`)

Executed:

1. **Renamed** the agent folder: `hr_default` → `hr_learning_and_development`
2. **Updated manifest** — added `team_id: 3864` to the seed_scope,
   changed slug + name to `hr_learning_and_development` /
   "HR / Learning & Development Agent".
3. **Rewrote the master prompt** to be L&D-specific instead of generic
   HR. Priorities: training programs, learner progress, content
   authoring, instructor ops, compliance training, LMS operations,
   effectiveness feedback.
4. **Rewrote the readme** with the corrected scope, routing
   precedence explanation, and override SQL examples.
5. **Deleted the stale DB row** (`id=1, slug=hr_default`).
6. **Re-ran bootstrap** — new row seeded: `id=2,
   slug=hr_learning_and_development, cat=4554, team=3864`.
7. **Re-verified routing**:
   - Meetings with `team=3864` → new agent ✓
   - Meetings with any other team inside HR (e.g. `team=9999`) → fall
     through to legacy path ✓ (as expected — those teams don't have
     agents yet)

### Why this matters

The pilot now genuinely models "one agent per team", not "one agent
per category masquerading as a team agent." Other HR teams (Recruiting,
Payroll, etc.) each get their own folder when we're ready — one row
per team, precedence rules take care of the rest.
